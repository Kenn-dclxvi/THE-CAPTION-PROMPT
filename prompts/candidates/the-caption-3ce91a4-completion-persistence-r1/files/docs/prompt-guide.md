# プロンプトガイド

## この文書の位置づけ
この文書は、なぜ `AGENTS.md` とロールプロンプトを分けるのかを説明するガイドである。

AI 駆動開発プロセスの正本は `docs/orchestration-process.md` である。このガイドは、その正本を繰り返すのではなく、曖昧依頼をどう扱うか、どの境界を AI が作ってはいけないかを補足する。

## 役割分離
- `AGENTS.md` は、親エージェント用のオーケストレーション制御である。
- `docs/orchestration-process.md` は、AI 駆動開発プロセスの仕様である。
- `prompts/plan.md` は、指示書作成用の自己完結プロンプトである。
- `prompts/audit.md` は、監査SA用の自己完結プロンプトである。
- `prompts/review.md` は、レビューSA用の自己完結プロンプトである。

この分離の目的は、親エージェントが境界と実装主体を管理し、独立性が必要な各ロールプロンプトが自分の責務だけを持つようにすることである。実装は専用ロールに固定せず、親が直接行うかSAへ委任するかをモデルが作業単位ごとに選ぶ。

## AI最適化の設計理由
この分業は、人間の組織図の写し取りではない。人間の組織は、人件費・人数・疲労・エゴという人間固有の制約を節約するために進化した形である。AI にはその制約がなく、別の失敗様式を持つ。この分業は、AI に存在しない制約を捨て、AI 固有の失敗様式に向けて作り直した統治の形である。

設計理由は次の8点である。各点の運用仕様は `docs/orchestration-process.md`、常時境界は `AGENTS.md` に置き、ここでは理由だけを示す。

1. 失敗様式で分ける。役割は人間の職種の写しでなく、AI が起こす失敗様式で分ける。スコープが狭いほど、エージェントが自分の出力を正当化する余地が消える。
2. 主張でなく成果物で照合する。検証は所感でなく、嘘をつけない成果物（テスト結果 / 差分 / grep 一致 / 型検査）への照合に寄せる。`scripts/check-placement.sh` はこの原理の一例である。
3. 情報の非対称性を制御レバーにする。各 SA が何を見て何を見ないかを設計で決める。監査には実装者の理由説明を渡さず、差分・仕様・テスト結果だけ渡す。説明されると追認してしまう弱点を構造で殺す。
4. AI 固有の病理へ敵対的に防御する。守る相手は人間の不注意でなく、確証バイアス・迎合・reward hacking である。テスト自体が改竄されていないか（`# noqa`・skip・xfail の追加、失敗パスの mock 握りつぶし、assertion 緩和）を一次チェックに含める。
5. 契約riskとquality riskを分ける。契約、authority、scope、invariantの照合は監査へ、runtime correctness、利用者影響、state safetyの確認はレビューへ渡し、必要な責務だけを独立workerへ起動する。
6. 親の希少資源はコンテキストと信頼である。実装の直接実行とSA委任は、固定された役職ではなく、作業の独立性・並列性・必要なコンテキストに応じてモデルが選ぶ。親の中核責務は、どの SA に何を見せるかのコンテキスト経路設計と、証拠が揃うまで次段へ通さない関所強制の二つに収束する。
7. 独立検証はriskに対応させる。required machine validationとdrift確認は常に維持し、machine evidenceだけでdoneを判定できないriskがある場合だけ監査 / レビューを追加する。変更量はriskの代用にしない。
8. 停止を観測事実へ結び付ける。実行開始後の未完了状態や手段への逡巡を新しい停止条件へ変換せず、固定済みの停止条件が成立した場合だけ止める。これにより、境界を広げずに完了条件まで進む。

人間向けの習熟順では、段階0-2 は組織比喩で直感を作り、段階3-4 でこの AI 固有の規律へ移行する。詳細は部門の習熟指針『AIエージェント活用指針』に置く。

## 基本方針
AI 駆動開発では、次の順序を守る。

```text
境界
委任
実行
```

### 境界
まず、作業境界が明確かを確認する。

境界とは、AI が作業するために必要な前提である。

- 目的
- 問題
- 対象
- 範囲
- 品質基準
- 完了条件
- 判断基準
- 仕様
- 構成方針
- 実行許可

これらが未定義で、AI が補完しないと進めない場合は停止する。

### 委任
ユーザーが明示的に委任した判断は、その範囲で非破壊に仮決定してよい。

ただし、委任は無制限ではない。AI ができるのはあくまで仮決定であり、実行は承認済み実装指示書、明示範囲、停止条件に従う。

### 実行
実行は、承認済み実装指示書または作業単位化した依頼の範囲内でのみ行う。

実行時は、明示された範囲内で、最小限・局所的・可逆的に行う。

## 原則
- AI は、問題・対象・範囲・完了条件・品質基準・実行許可を補完しない。
- 曖昧な依頼を、調査・探索・確認・テスト実行・探索的修正に読み替えない。
- 推定は、前提・候補・方針の提示にのみ使い、実行許可・変更範囲・完了条件・品質基準には変換しない。
- 推定で、問題定義・削除・移動・構成変更・タイポ補正を決めない。
- 半具体的な変更依頼は即実装しない。

制御対象は、推論をそのまま実行許可・変更範囲・完了条件・品質基準に変換してしまうことである。

```text
推論は使ってよい。
ただし、推論を実行境界に変換しない。
```

## AGENTS.md に残す制御
`AGENTS.md` は、常時適用される実行境界を定義する。

`AGENTS.md` に残すべきものは、破ると副作用・誤実行・スコープ逸脱につながる制御である。

- 承認済み実装指示書または明示された作業単位化条件がある場合に限り、実装境界を固定して実行する。
- 問題、対象、範囲、完了条件、品質基準、実行許可を補完しない。
- 最新ユーザー入力は承認済み実装指示書と矛盾しない範囲でのみ扱う。
- 親エージェントは実装、修正、テスト実行を直接行うかSAへ委任するかを選べるが、監査、selected routeが要求する PRレビュー相当の品質確認は直接行わない。
- 条件付き事前承認は、明示条件を満たすことを推定なしで確認できる場合だけ作業単位化条件として扱う。
- 条件付き事前承認テンプレートと既定テスト定義は、IDと条件を一意に照合できる場合だけ使う。
- 実装開始前に実装後確認経路を固定し、required machine validation成功後にrouteが要求する監査SA / レビューSAだけを起動する。
- 監査SAとレビューSAの指摘分類を親が管理する。
- 自動再修正は、承認済み範囲に直接対応する停止指摘と、承認済み範囲内かつ最小修正可能な重大指摘に限る。

## プロンプト設計の抽象化原則
プロンプト本体は、事故パターンを列挙する場所ではない。

プロンプト本体には、事故を防ぐための抽象化された境界条件を書く。

新しいルールを追加する前に、次を確認する。

1. これは事故例か、原則か。
2. もっと上位概念で言えるか。
3. 対象列挙ではなく境界定義にできるか。
4. 禁止ではなく成立条件で書けるか。
5. 契約 / ガイド / テストのどこに置くべきか。
6. 既存ルールと同じ意味なら統合できるか。
7. そのルールを削ってもテストで検出できるか。
8. これは成立条件か、手続きか。手続き・列挙・出力様式なら正本へ送る。

### 配置規律
記述の置き場所は3層に固定する。

- `AGENTS.md` は常時適用の実行境界のみを置く。1概念につき成立条件1行(成立しなければ停止)に畳み、構成項目の逐語列挙・照合手順・展開手順・出力フォーマットは置かない。共有する構成項目は冒頭の用語定義で1度だけ定義し、本文は用語で参照する。
- `docs/orchestration-process.md` は正本とする。構成項目の列挙・照合フロー・展開手順・出力様式はここにのみ置き、`AGENTS.md` からは概念名で参照する。
- `docs/prompt-guide.md` は設計理由と配置の不変条件を置く。

不変条件は以下とする。

- 構成項目の逐語列挙は正本または `AGENTS.md` 冒頭の用語定義にのみ置き、`AGENTS.md` 本文には書かない。
- 同一概念の成立条件を `AGENTS.md` 本文の複数セクションに重複定義しない。
- 新ルール追加時は上記チェックリストを適用し、手続き・列挙・出力様式は正本へ送る。

この不変条件は機械検査できる。`scripts/check-placement.sh` が root と overlay の `AGENTS.md` 本文への構成項目の逐語列挙の再混入を grep で検出する。レビューSAの観点または CI に組み込む。

## 非破壊
- 「分析して」「レビューして」「確認して」「案を出して」「比較して」は、明示対象に限り非破壊として扱う。
- 非破壊では、関連文書・既存コード・差分を読んでよい。
- 明示された文書・差分の非破壊レビュー依頼は、親エージェントが直接扱える。
- selected routeが要求する PRレビュー相当の品質確認工程はレビューSA専管であり、親エージェントは直接行わない。
- 未指定の問題探索・修正対象化・実行完了には進めない。

非破壊は、読むことと整理して返すことまでを許可する。

## 停止
- 問題・対象・範囲・完了条件が未定義の依頼は、調査・実行・修正に進めず、不足項目を返す。
- AI が対象・場所・目的・完了条件を選ぶ必要がある作成・変更は実行しない。
- 「必要」「適切」「いい感じ」などの曖昧語が判断を左右する場合は、AI が基準を作らない。
- 必要最小限の確認1回で境界が定まる場合は、その確認だけ返す。
- 承認済み実装指示書がない場合は、実装へ進まない。
- 実装・変更の意図がありながら境界が未確定の依頼は、停止に代えてプランSAで非破壊の草案を作る。草案は実行境界を生成せず、境界を推定なしで確定できなければプランSAが不足項目を返すため、補完禁止と両立する(`docs/orchestration-process.md` のケースE)。

停止とは、次のような実行行為に進まないことである。

- 調査
- 探索
- 確認
- テスト実行
- ファイル作成
- 修正
- 削除
- 移行
- 完了報告

一方で、非破壊の整理や、境界を1回で定めるための最小確認は許可される。

## 実行許可
重要なのは、次の区別である。

```text
作成したい = 意図表明
作成して = 実行許可
```

同様に、次も区別する。

```text
進めたい = 意図表明
進めて = 直前方針への承認
```

「進めて」は、直前に提示した方針への承認としてのみ扱う。この場合に実行してよいのは、直前方針に含まれていた範囲だけである。

ユーザーは、草案が明示条件を満たすなら追加確認なしで作業単位化して実行してよい、という条件付き事前承認を先に宣言できる。この場合も、条件・対象・範囲・完了条件・品質基準・実行許可を AI が補完してはならない。条件充足を推定で判断する必要がある場合は停止する。`commit` / `push` / `deploy` / 外部送信は、別途明示許可がない限り条件付き事前承認に含めない。

条件付き事前承認テンプレートは、実行許可を強める仕組みではなく、明示済みの条件をIDで参照する仕組みである。テンプレートIDがあっても、条件・対象・範囲・完了条件・品質基準・実行許可・除外操作のいずれかを AI が補完する必要がある場合は実行へ進まない。

既定テスト定義も同じく、テスト条件を推定する仕組みではない。既定テストIDと適用条件を推定なしで照合できる場合にだけ、展開済みのテストコマンドまたは `なし（根拠）` を作業単位の `tests` として扱う。コード変更を含む作業単位では、既定テスト定義を使う場合でもテストコマンド必須である。

## リポジトリ参照
プロジェクト固有の参照先は、この汎用プロンプトセットへ直接埋め込まず、適用先リポジトリのオーバーレイまたは明示入力に分離する。

参照文書は、AI が自由に問題を探すための入口ではない。参照文書は、依頼済みの作業境界を確立または検証するために使う。

## 書き味ポリシー
このガイドでいう AI制御文書は、`AGENTS.md` / `prompts/*.md` / `docs/orchestration-process.md` / `docs/prompt-guide.md` である。

### 用語規範
- 一概念一語を優先する。
- 略語を使う場合は、文書冒頭で定義してから使う。
- `AGENTS.md` 冒頭の略語定義は `AGENTS.md` 内でのみ有効とし、他文書へ波及させない。
- ロールプロンプトは他工程を工程名で参照し、役割名は調整主体向け文書で使う。

### 記号・空白規則
- 強調スペースを使わない。
- 数値と助数詞は続け書きする。
- コードスパンと和文の間は半角スペース1つとする。ただし句読点・括弧・中黒などの全角記号と隣接する場合はスペースを置かない。
- スラッシュ列挙は「A / B / C」とする。
- ファイル名・パス・ディレクトリ・コマンド・キー名はコードスパンで書く。
- 空行は最大1行とする。

### 文体の型
- ロールプロンプト冒頭の役割セクション第1項目には、何をするかを書く。
- 文の箇条書きは終止形と句点で書く。
- 名詞列挙は句点なしの体言止めで書く。
- ロールプロンプトの標準セクション順は、役割→入力→ルール→出力を基本骨格とする。

### 自己完結性
ロールプロンプトが持ってよい知識は以下に限る。

1. 自分の責務・禁止操作・停止条件
2. 自分への入力の形式と前提条件
3. 自分の出力の形式
4. 自分が読んでよいリポジトリ文書の限定列挙と読取条件
5. 自分の判定基準で使用する分類名とその定義

ロールプロンプトが持ってはならない知識は以下である。

1. 他SAの責務・役割の定義
2. 自分・他SAの起動順序・起動条件
3. ループ制御・回数管理・完了判定・自動修正の成立条件の定義
4. 他SAのロールプロンプトのパス。ただしプランSAは指示書設計に必要な範囲で例外とする。
5. 自分が産出しない分類体系の定義

## 成功条件
- 明示された境界内だけで作業している。
- 実行理由が、ユーザー指示・直前の観測事実・既存コード・仕様・テストに束縛されている。
- 無関係な変更・調査・確認を含まない。

成功とは、単にテストが通ることではない。AI が明示された境界内だけで作業し、勝手な解釈や拡張をしていないことである。
